# 统一数据库访问接口以消除散落 SQL

_来源：6a45874 → ea5470e 提交周期内记录的编码计划——内容为规划时意图，实现可能滞后或有出入。_

**状态：** accepted

## 背景
业务代码（特别是 orchestrator.py）中散落了 37 处直接 SQL 语句，导致数据访问逻辑与业务逻辑耦合，难以维护且容易引发 SQL 注入或 schema 变更时的广泛修改。

## 决策驱动
- 数据访问层抽象
- 代码复用
- 安全性与维护性

## 备选方案
- **继续使用内联 SQL** _（已否决）_ — 优点：无需额外封装，直观看到查询逻辑；缺点：SQL 散落在各处，重复代码多，Schema 变更时需修改多处，缺乏统一类型检查
- **在 database.py 中封装所有数据访问函数** — 优点：集中管理 SQL，提供统一接口，便于添加日志、监控和类型注解，业务层只调用方法；缺点：初期工作量较大，需逐一替换并验证行为一致性

## 决策
将所有散落在业务代码中的 SQL 迁移至 database.py 中的专用封装函数（如 increment_brain_think_count, check_existing_deliberation 等），遵循“先封装、后替换”的原则，确保生产环境稳定性。

## 影响
实现了数据访问层的统一入口，消除了代码重复，提升了 SQL 管理的集中度。未来修改数据库 Schema 或优化查询性能时，只需修改 database.py 中的对应函数。同时为后续补充类型注解奠定了基础。
