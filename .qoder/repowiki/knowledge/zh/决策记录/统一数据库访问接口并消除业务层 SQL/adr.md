# 统一数据库访问接口并消除业务层 SQL

_来源：76b5166 → 6a45874 提交周期内记录的编码计划——内容为规划时意图，实现可能滞后或有出入。_

**状态：** accepted

## 背景
业务逻辑文件 orchestrator.py 中散落着 37 处直接 SQL 语句，导致数据访问逻辑与业务逻辑耦合，且缺乏统一的类型安全和维护入口。

## 决策驱动
- 关注点分离
- 代码可维护性
- 类型安全基础

## 备选方案
- **保留业务层直接 SQL** _（已否决）_ — 优点：无需额外封装，执行路径短；缺点：SQL 分散难以维护，无法统一进行类型注解或优化，违反分层架构原则
- **在 database.py 中封装所有数据访问** — 优点：业务层仅调用封装函数，SQL 集中管理，便于后续补充类型注解和优化；缺点：需要编写大量封装函数，初期工作量较大

## 决策
将所有散落在 orchestrator.py 中的直接 SQL 迁移至 database.py 中的专用封装函数（如 increment_brain_think_count, check_existing_deliberation 等）。遵循“先封装、后替换”的原则，每次替换后验证生产稳定性。

## 影响
实现了数据访问层与业务逻辑层的解耦，为后续全面补充类型注解奠定了基础。业务代码不再直接依赖 SQL 字符串，提高了代码的可读性和安全性。
