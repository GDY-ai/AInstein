# 将编排器拆分为模块化子组件

_来源：6a45874 → ea5470e 提交周期内记录的编码计划——内容为规划时意图，实现可能滞后或有出入。_

**状态：** accepted

## 背景
核心文件 orchestrator.py 膨胀至 3848 行，严重违反单一职责原则，导致维护困难且容易引入 Bug（如已发生的 NameError 和并发问题）。需要在不影响生产运行的前提下进行结构性重构。

## 决策驱动
- 代码可维护性
- 单一职责原则
- 降低耦合度

## 备选方案
- **保持单体大文件并仅做内部函数整理** _（已否决）_ — 优点：改动风险极低，无需处理模块间导入依赖；缺点：无法根本解决文件过大导致的认知负荷和维护难题，难以并行开发
- **按功能领域拆分为 orchestrator/ 包结构** — 优点：清晰分离关注点（策略、工具提案、主脑协调、博弈触发），每个模块规模可控（~200-400行），便于独立测试和维护；缺点：需要谨慎处理模块间依赖和状态共享，迁移过程需分步验证以防回归

## 决策
采用按功能领域拆分方案，将 orchestrator.py 重构为 orchestrator/ 包，包含 __init__.py (主类), strategy.py, tool_proposal.py, master_coordinator.py, deliberation_trigger.py 和 constants.py。迁移顺序定为：常量 -> 工具提案 -> 主脑协调 -> 策略 -> 博弈触发，以确保依赖正确建立。

## 影响
代码结构显著清晰，各功能模块职责单一。后续新增功能只需修改对应子模块，降低了合并冲突概率。但引入了模块间调用的开销，且主类 ATAOrchestrator 需正确初始化和管理这些子组件的生命周期。
